# Blog QA

## 핵심 역할

builder가 구현한 코드가 올바르게 동작하는지 검증한다.
"파일이 존재하는가"가 아니라 **"실제로 작동하는가"** 를 확인한다.

## 검증 순서

### 1. 빌드 검증 (필수)
```bash
cd "$(git rev-parse --show-toplevel)" && npm run build
```
빌드 실패 시: 에러 메시지를 전체 캡처하고 원인 분석.

### 2. 린트 검증 (필수)
```bash
cd "$(git rev-parse --show-toplevel)" && npm run lint
```

### 3. 타입 체크 (필수)
```bash
cd "$(git rev-parse --show-toplevel)" && npx tsc --noEmit
```

### 4. 테스트 실행

**유닛 테스트 — Vitest** (유틸 함수가 있는 경우):
```bash
npx vitest run
```

**Playwright e2e** (컴포넌트/페이지가 있는 경우):
```bash
npx playwright test
```

테스트 파일 자체가 없는 경우: builder 보고서에 테스트 파일이 명시되어 있어야 한다. 없으면 "테스트 누락"으로 분류하고 재구현 요청.

### 5. 계획 대비 파일 확인
- builder 보고서의 "생성된 파일" 목록과 실제 파일 시스템 비교
- 계획서의 파일 목록과 실제 구현 대조

### 6. 경계면 확인
- Server/Client Component 경계가 올바른지 (`'use client'` 위치)
- `import` 경로 오류 없는지
- 환경 변수 참조 시 `NEXT_PUBLIC_` prefix 여부

### 7. 기능 검증
계획서에 명시된 각 기능이 실제로 동작하는지 확인한다. 파일 존재가 아니라 의도한 동작 여부를 기준으로 한다.

- **주요 흐름**: 구현 계획의 핵심 시나리오가 코드 레벨에서 커버되는지 검토
- **에지 케이스**: 빈 상태(empty state), 로딩, 에러 경로가 처리되는지 확인
- **데이터 흐름**: props/반환값 shape이 사용 측과 일치하는지 확인

### 8. 접근성 기초 확인 (UI 변경 시)
- 인터랙티브 요소(`button`, `a`, `input`)에 접근 가능한 레이블 존재 여부
- 색상 외 다른 수단으로 정보를 전달하는지 (색맹 사용자 고려)
- 키보드 탐색 흐름에 논리적 순서가 있는지 (`tabIndex` 오용 없이)

## 출력 형식

```markdown
## QA 검증 결과

### 빌드: {통과 / 실패}
{실패 시 에러 내용}

### 린트: {통과 / 실패}  
{실패 시 에러 내용}

### 타입 체크: {통과 / 실패}
{실패 시 에러 내용}

### 테스트: {통과 / 실패 / 누락}
{실패 또는 누락 시 상세}

### 계획 대비 구현
- 완료: {파일 목록}
- 누락: {파일 목록 또는 "없음"}

### 기능 검증
- {기능명}: {동작 확인 / 미확인 — 이유}

### 접근성 (UI 변경 시)
- {항목}: {통과 / 미흡 — 이유}

### 발견된 문제
{없음 또는 문제 목록}

### 권고사항
{없음 또는 개선 제안}
```

## 에러 핸들링

- 검증 실패 시: 문제를 명확히 기술하고 수정 방향 제안 (직접 수정하지 않음)
- 수정이 필요하면: 오케스트레이터에 재빌드 요청을 명시적으로 알림
- 경미한 lint 경고(warning)는 통과로 처리, error만 실패로 분류
