# 12. 목록 페이지네이션 전략

Date: 2026-06-17

## Status

Accepted (조건부 — 목록이 임계치를 넘어 페이지네이션을 도입할 때 적용)

> **현재 상태:** 글 수가 적어 목록(`/writing` 등)은 **분할 없이 전체 노출**한다. 페이지네이션은 도입하지 않은 상태이며, 찾기는 태그·아카이브·검색이 담당한다. 목록 한 페이지 로딩이 무거워지는 시점(대략 글 30~50편 초과)에 이 ADR의 방식으로 전환한다. 본 ADR은 "그때 어떻게 할지"를 미리 못박는 standing decision이다.

## Context

글·태그·아카이브 등 목록 라우트가 임계 건수를 넘으면 긴 목록을 어떻게 탐색하게 할지 결정해야 한다. 개별 글 URL(`/writing/[slug]`)은 슬러그 기반으로 이미 확정돼 있으며, 이 결정은 **목록 탐색 방식과 그 URL**에만 해당한다. 이 블로그는 정적 생성(SSG) 기반이고 검색 엔진 노출이 목표라는 점이 모든 판단의 전제다.

### 목록 탐색 방식 후보

- **분할 없음 (전체 노출)** — 한 페이지에 모든 항목. 글이 적을 때 가장 단순하고 SEO에 유리하나, 항목이 많아지면 페이로드·렌더 부담이 커진다. → **현재 채택(임계치 전까지)**
- **번호 페이지네이션** — 목록을 N개 단위로 끊어 페이지별 URL을 부여한다.
- **더보기 버튼 / 무한스크롤** — 추가 항목을 스크롤·클릭으로 점진 로드한다.

### 더보기 / 무한스크롤을 채택하지 않는 이유

- **순수 클라이언트 방식**은 첫 로드 이후 항목이 JS로 주입되므로, 크롤러가 2페이지 이후 글에 도달하지 못해 **SEO 색인에 불리**하다. SSG·검색 노출 목표와 정면으로 충돌한다.
- **SEO를 지키려면** 전체 항목을 HTML에 렌더해두고 더보기는 이미 있는 DOM을 시각적으로만 펼치는 방식이어야 하는데, 그러면 초기 페이로드가 전체와 같아 "대량 목록 분할"이라는 목적을 달성하지 못한다(사실상 *분할 없음*과 동일).
- 무한스크롤은 푸터 도달 불가·"현재 위치" 상실 등 접근성·길찾기 문제도 동반한다.

따라서 **대량 목록을 실제로 분할해야 하는 시점**에는 번호 페이지네이션이 적합하다. 더보기/무한스크롤은 SEO가 중요하지 않은 부차적 목록에 한해 향후 개별 검토한다.

### 번호 페이지네이션의 URL 방식

번호 페이지네이션을 쓸 경우 URL 방식 후보는 다음과 같다. 페이지네이션 URL은 한번 공개되면 크롤러·외부 링크가 참조하므로 나중에 바꾸기 어렵다.

- **쿼리스트링** — `/writing?page=2`
- **경로 세그먼트** — `/writing/page/2`

쿼리스트링 방식은 SSG와 충돌한다. Next.js App Router에서 `?page=N`은 정적 경로로 잡히지 않아 별도 동적 처리가 필요하고, 크롤러가 `?page=2`를 동일 페이지의 파라미터로 보거나 canonical 처리가 모호해질 수 있다.

## Decision

목록 탐색은 현재 **분할 없음**으로 두고, 분할이 필요해지면 더보기/무한스크롤이 아닌 **번호 페이지네이션**을 도입한다(근거는 Context). 그 페이지 URL은 **경로 세그먼트 방식** `{목록}/page/{n}` 을 채택한다.

- 형식: `/writing/page/2`, `/tag/[tag]/page/2`, `/archive/[year]/page/2` 등
- 1페이지는 베이스 경로(`/writing`)로 두고, `/writing/page/1`은 베이스로 301 리다이렉트한다
- canonical은 각 페이지 자신을 가리키며, 1페이지의 canonical은 베이스 경로다
- 페이지 번호는 `generateStaticParams`로 빌드 시 정적 생성한다

적용 대상: 글·태그·아카이브·메모·책장의 목록 라우트.

## Consequences

긍정적 결과:

- 각 페이지를 정적 파일로 생성할 수 있어 SSG 파이프라인과 자연스럽게 맞는다
- 크롤러가 페이지마다 독립 URL로 인식하고, canonical·rel prev/next 등 SEO 신호를 명확히 제어할 수 있다
- URL이 사람이 읽기 좋고 공유에 적합하다

수용해야 할 결과:

- 페이지 수만큼 정적 경로가 늘어 빌드 산출물이 커진다(개인 블로그 규모에선 무방)
- 번호 기반 페이지네이션의 본질적 한계로, 글이 추가·삭제되면 특정 `/page/n`이 가리키는 콘텐츠가 달라진다. 영구 링크 용도로는 부적합하나 목록 탐색용으로는 허용한다
- 중복 색인을 막기 위해 `/page/1` → 베이스 301 리다이렉트를 반드시 정의해야 한다
