# 4. 검색 구현 방식

Date: 2026-06-15

## Status

Accepted

## Context

블로그 검색 기능을 어떻게 구현할지 결정해야 한다. 주요 후보는 다음과 같다:

- **클라이언트 사이드 검색 (fuse.js)** — 글 목록을 클라이언트에 내려받아 브라우저에서 검색
- **서버 사이드 검색** — 검색 요청마다 서버에서 처리
- **외부 검색 서비스** (Algolia, Typesense 등) — 전용 검색 인프라 사용

정적으로 생성되는 블로그이므로 별도 서버를 운영하지 않으며, 개인 블로그 특성상 글 수가 많지 않다.

## Decision

**클라이언트 사이드 퍼지 검색 (fuse.js)** 을 채택한다. 검색 대상은 본문 전체가 아닌 제목·태그·요약 필드로 제한한다.

## Consequences

긍정적 결과:
- Algolia 같은 외부 검색 서비스 없이 동작하므로 비용과 의존성이 없다
- 정적 블로그에서 검색 API 서버를 별도로 운영할 필요가 없다
- fuse.js는 오타 허용 검색(퍼지 매칭)을 지원하여 사용 편의성이 높다

수용해야 할 결과:
- 글 수가 수천 개를 넘으면 초기 데이터 로딩 부담이 커질 수 있다 (현재는 문제없음)
- 본문 전체를 인덱싱하면 페이로드가 커지므로 검색 대상을 제목·태그·요약으로 제한한다
- Algolia 등 전용 서비스에 비해 검색 정확도와 기능(동의어, 랭킹 등)이 부족하다
