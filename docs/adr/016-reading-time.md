# 16. 읽기 시간 계산 방식

Date: 2026-06-18

## Status

Accepted

## Context

글 상세 페이지에 "읽는 데 약 N분"을 표시한다(PRD 콘텐츠 > 글 상세, 우선순위 상). 이 값을 어떻게 계산할지 결정해야 한다.

초기 구현은 `raw 문자열 길이 ÷ 500`(분당 500자)이라는 단순 추정이었으나 세 가지 문제가 있다.

- **코드 블록·마크다운 문법·JSX·URL이 글자 수에 그대로 포함**되어, 코드가 많은 글일수록 시간이 과대 추정된다. 이것이 가장 큰 오차원이다.
- 한국어와 영어를 구분하지 않는다. 영어는 글자가 아니라 단어 단위로 읽으므로 글자 기반 추정이 부정확하다.
- `500`이라는 상수의 근거 출처가 없다.

선택에 앞서 "이 블로그에서 읽기 시간이 무엇을 위한 것인가"와 "무엇이 적합 기준인가"를 먼저 정리했다.

### 판단 기준

1. **목적 — 정밀도가 아니라 기대치 신호.** 읽기 시간은 "이 글 7분이면 읽겠네" 수준의 어림수다. ±1~2분 오차는 무해하고, 배수로 틀리는 것이 실패다. 이미지별 시간까지 재는 정밀 모델은 오버엔지니어링이다.
2. **콘텐츠 정합성.** 콘텐츠는 한국어 산문 + 코드 블록 + 영어 용어 혼합이다. 한국어를 글자/음절 또는 어절 단위로 적절히 셀 수 있어야 하고, 코드 블록을 산문과 다르게(또는 제외) 처리해야 한다.
3. **정확도의 실제 레버.** 최대 오차는 CPM 상수값이 아니라 코드/마크다운을 본문에서 걷어내는 전처리 여부다. 이 MDX 전처리는 어떤 외부 라이브러리도 대신 해주지 않으므로 우리가 해야 한다.
4. **의존성 비용.** ADR-011 및 프로젝트 원칙은 "기존 패키지 우선, 새 의존성 신중"이다. 핵심 로직이 사실상 `글자수 ÷ 상수`다.
5. **투명성.** 근거 없는 매직 넘버를 피하고, 인용 가능한 출처에 기반해야 한다.
6. **정제 강건성과 계산 위치 (실질적 핵심).** 라이브러리든 커스텀이든 정제(코드/마크다운 제거)는 공통 과제이므로 선택을 가르는 요소가 아니다. 진짜 결정은 *정제를 어디서·어떻게 하느냐*다 — (a) 컴파일하지 않은 raw MDX를 정규식으로 정제할 것인가, (b) 컴파일된 텍스트(렌더 결과)에서 정제할 것인가. 정확도·강건성·유지보수 위험이 전부 이 지점에 모인다. 글 목록(`getPostMetaList`)은 성능을 위해 본문을 컴파일하지 않는 경량 파서이므로, 목록과 상세에서 동일한 읽기 시간을 보장하려면 계산 위치를 일관되게 정해야 한다.

### 후보

- **Wolfram `TimeToRead`** — 순수 WPM(분당 단어 수) 모델. 영어 238, 독일어 179 등 12개 언어 상수. 단어 경계가 공백 기반이라 **한국어/CJK를 지원하지 않는다**(기준 2 미충족).
- **`reading-time`** (ngryman, Medium 방식) — npm de-facto 표준(주간 다운로드 약 78만 회). CJK는 Unicode 범위로 글자 수를 세고 알파벳 단어는 별도 카운트, 기본 200 WPM. 다만 최신 1.5.0이 2021-09 배포로 갱신이 멈췄고, 국제화(출력 문자열) 지원이 없다.
- **`reading-time-estimator`** — `reading-time`의 국제화·CJK 강화판. 주간 다운로드 약 1.4만 회, 최신 2.6.0(2026-04 배포)으로 비교 후보 중 유일하게 최근까지 활발히 유지보수된다. 소스(2.6.0)를 확인한 결과 토큰화 정규식이 **한글 음절 블록(`가–힯`)을 라틴 문자와 같은 `+` 클래스에 두어, 한국어를 어절(공백 구분 단위)로 카운트**한다(중국어·일본어 표의문자는 글자 단위). 즉 한국어 어절과 영어 단어가 같은 토큰 단위로 합산되어 **단일 WPM(기본 200)이 한/영 혼합 콘텐츠에 자연스럽게 맞는다**(한국어 약 200~250어절/분 ≈ 영어 200단어/분). 내부적으로 `marked`로 마크다운을 파싱하나, 코드 블록 텍스트는 카운트에서 제외되지 않아 **코드 strip은 우리 몫**이다. `ko` 출력 로케일은 없으나 `minutes`만 사용하고 라벨은 직접 렌더하면 무관하다.
- **`read-time-estimate`** — 한국어/중국어/일본어 글자당 속도를 개별 적용하고 이미지 시간까지 계산한다. 한국어 정밀도는 가장 높으나, 주간 다운로드 약 3.6천 회로 비교 후보 중 가장 적고 최신 버전이 0.0.3(2021-01 이후 갱신 없음)이라 유지보수가 사실상 중단됐다. 또한 정밀도가 목적(기준 1)에 비해 과하다.
- **커스텀 유틸 + 출처 명시** — 의존성 0, 전 과정 완전 제어. 단점은 어절/CJK 토큰화(결합 문자·다국어·어절 분리 등 엣지가 많음)와 정제 로직을 모두 직접 유지보수해야 한다.

정제(코드 strip)는 어떤 라이브러리도 대신 해주지 않아 모든 옵션에서 우리 몫이다(기준 3·6). 따라서 진짜 비교 지점은 "토큰화·계산을 직접 짤 것인가, 검증된 라이브러리에 위임할 것인가"다. 어절/CJK 토큰화는 결합 문자·다국어·어절 분리 등 엣지가 많아 직접 짜면 미묘한 버그가 생기기 쉬운 영역인데, `reading-time-estimator`는 이를 한국어 어절 모델로 올바르게 처리하고(단일 200 WPM이 한/영에 모두 적합) 활발히 유지보수되며 계산은 빌드 타임 서버에서만 돌아 클라이언트 번들에 영향이 없다. 그렇다면 **토큰화·계산은 라이브러리에 위임하고, MDX 고유의 코드 strip만 우리가 맡는 책임 분담**이 가장 적합하다.

## Decision

**책임 분담** 방식을 채택한다. 토큰화·분 계산은 **`reading-time-estimator`** 에 위임하고, MDX 고유의 전처리(코드 strip)는 우리 유틸(`src/lib/reading-time.ts`)이 담당한다.

흐름:

1. **우리 유틸이 raw MDX(frontmatter 제거 후)에서 코드를 제거**한다 — 펜스 코드 블록(```` ``` ````)과 인라인 코드(`` ` ``)를 우선 제거. 코드는 산문 읽기 속도로 읽히지 않아 가장 큰 왜곡원이고, 라이브러리가 코드 텍스트를 카운트에서 빼주지 않으므로 우리가 한다. 펜스/백틱은 마크다운에서 가장 안정적인 문법이라 정규식이 깨질 위험이 낮다.
2. 정제된 텍스트를 **`readingTime(text, { wordsPerMinute: 200 })`** 에 넘겨 `minutes`를 얻는다. 라이브러리가 한국어를 어절로, 영어를 단어로 같은 단위로 토큰화하므로 단일 200 WPM이 한/영 모두에 적합하다.
3. **"N분" 라벨은 직접 렌더**한다(`ko` 출력 로케일 부재). 라이브러리 결과 중 `minutes`만 사용한다.

**계산 위치.** 목록(`getPostMetaList`)이 본문을 컴파일하지 않는 경량 파서이므로, 목록·상세 모두 위 raw 기반 흐름을 동일하게 써서 값이 일치하도록 한다.

상수 출처: 영어 200단어/분은 `reading-time`·`reading-time-estimator`의 통용 기본값이다. 한국어는 어절 단위로 카운트되며 성인 묵독 약 200~250어절/분(≈500~600음절/분)으로, 같은 200 WPM 상수로 근사된다.

## Consequences

긍정적 결과:

- 어절/CJK 토큰화(결합 문자·다국어·어절 분리 등 엣지가 많은 영역)를 검증·유지보수되는 라이브러리에 위임해, 직접 구현 시 생길 미묘한 카운팅 버그를 피한다.
- 한국어=어절, 영어=단어가 같은 토큰 단위로 합산되어 단일 200 WPM이 혼합 콘텐츠에 자연스럽게 맞는다.
- 코드 블록을 카운트에서 제외해 코드 많은 기술 글의 과대 추정을 해소한다 — 정확도의 최대 레버를 우리가 직접 통제한다.
- 읽기 시간 계산은 빌드 타임(SSG) 서버에서만 실행되어 `reading-time-estimator`(및 전이 의존성 `marked`)가 **클라이언트 번들에 포함되지 않는다**.
- 채택 근거와 상수 출처가 본 ADR에 문서화되어 매직 넘버 문제가 해소된다.

수용해야 할 결과:

- 새 의존성 1개(`reading-time-estimator`)가 추가된다. ADR-011의 "새 의존성 신중" 원칙에 따른 판단이며, 활발한 유지보수·서버 전용(번들 무영향)·토큰화 위임 가치를 근거로 수용한다.
- **코드 strip 정제 로직은 여전히 우리 몫이며 직접 유지보수한다.** 펜스·인라인 코드라는 안정적 문법으로 범위를 한정하고 단위 테스트로 회귀를 방어한다. 단, 코드 외 MDX/JSX 문법이 본격적으로 늘면 정제 규칙 갱신이 필요할 수 있다.
- 라이브러리의 토큰화·국제화 정책 변경에 종속된다. 한국어를 어절로 세는 현재 동작(2.6.0 소스 확인)에 의존하므로, 메이저 업그레이드 시 카운팅 동작 회귀를 테스트로 확인한다.
- 읽기 시간은 본질적으로 근사치이며, 독자·글 종류(산문/기술)·숙련도에 따라 실제 시간은 달라진다. 본 값은 기대치 신호일 뿐 정밀 측정이 아니다.
