# 13. 감상 항목 데이터 소싱과 저장

Date: 2026-06-17

## Status

Accepted

## Context

감상 항목(책·애니)을 어떻게 저장하고 메타데이터를 수집할지 결정해야 한다. 감상 항목은 cover·author·이미지 치수·장르·방영 상태 같은 구조화 메타가 많은데, 이를 손으로 채우긴 번거롭다. 외부 소스의 식별자만 있으면 나머지 메타는 자동 수집할 수 있다.

두 가지 요구가 충돌한다:

- 감상에 대한 **기록(prose)**을 자유롭게 남기고 싶다 — 구조화 데이터(JSON)에 본문을 섞기 어렵다.
- 메타는 외부 ID로 **자동 수집**하고 싶다 — 구조화된 필드 저장이 필요하다.

저장 방식 후보:

- **JSON 단일 데이터 파일** — 메타 저장엔 깔끔하나 본문(기록)을 담기 어렵다.
- **항목별 MDX** — frontmatter(메타) + 본문(기록)을 한 파일에.
- **JSON 메타 + 별도 review MDX** — 한 항목이 두 파일로 쪼개져 복잡하다.

## Decision

**항목별 MDX 파일**(`src/content/library/*.mdx`)로 저장한다. frontmatter는 구조화 메타, 본문은 선택적 감상 기록이다.

- 작성자는 `title`·`type`만 입력한다(`date`는 선택).
- fetch 스크립트가 **제목으로 외부 소스를 검색**해 외부 식별자(`id`)와 `cover`·`width`/`height`·`author`·`genres`·`status` 등을 frontmatter에 채운다. `id`는 단일 필드이며 `type`에 따라 해석된다 — `book`은 ISBN(Naver Book), `anime`는 AniList ID. 해석된 `id`는 frontmatter에 **write-back**해 다음 실행부터 정확 매칭으로 고정된다. `id`를 직접 적으면 제목 검색을 건너뛰는 override가 된다.
- 자동 필드의 정본은 외부 소스이며 frontmatter는 캐시다.
- 본문은 자유 형식 기록이며, 없으면 비워 둔다.
- 작성 마찰을 줄이기 위해 "파일 생성 + 메타 fetch"를 한 커맨드(`add`)로 묶는다.

유형을 책·애니로 한정한 것은 실제 기록 대상이 그 둘이기 때문이다(영화·만화는 제외).

JSON을 택하지 않은 이유:

- 본문(기록)을 깔끔히 담지 못하고 prose가 구조 데이터에 섞인다.
- 항목을 추가할 때마다 같은 파일이 바뀌어 **머지 충돌·노이즈 diff**가 생기고 PR 리뷰가 어렵다(docs-as-code 원칙과 상충).
- 쉼표·따옴표 하나만 틀려도 **전체 파일 파싱이 실패**한다 — 항목별 파일은 장애가 한 항목에 격리된다.
- `src/data`에 두면 글·메모(`src/content`)와 **저장 위치·작성 모델이 이원화**된다.

분리(JSON + 별도 MDX)는 한 항목이 두 곳으로 쪼개진다. 항목별 MDX는 메타와 기록을 한 파일에 두면서 글·메모와 저장 방식이 일관된다(ADR-002).

## Consequences

긍정적 결과:

- 메타(frontmatter)와 기록(body)이 분리되어 prose-in-JSON 문제가 사라진다
- 글·메모와 동일하게 MDX·git 친화적이다(항목별 diff, 리뷰 가능)
- 제목 기반 자동 수집으로 수동 입력이 `title`·`type` 두 가지로 최소화된다

수용해야 할 결과:

- 외부 API(Naver Book / AniList) 의존과 시크릿(`NAVER_CLIENT_ID`·`NAVER_CLIENT_SECRET`)이 필요하다(AniList는 키 불요). 빌드가 아니라 **작성 시점 스크립트**로 분리해 빌드 안정성을 지킨다
- 제목 검색은 모호할 수 있다(동명 도서, 시즌 여러 개 애니). 최선 매칭 ID를 frontmatter에 write-back해, 작성자가 확인·수정 후 재실행하면 정확 매칭으로 고정된다
- 자동 필드는 캐시이므로, 외부 변경(예: 애니 방영 `status`)을 반영하려면 스크립트를 재실행해야 한다
- 이미지 치수 측정용 `image-size` 등 스크립트 의존성 추가가 필요하다(감상 마일스톤에서 HITL로 확인)
- 항목 수만큼 파일이 늘어난다(글과 동일한 특성)
